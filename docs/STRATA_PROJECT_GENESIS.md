# Strata 项目诞生过程

**用途**:面试讲述用 / 简历附录 / 项目背景介绍
**时间跨度**:2026 年初 → 2026 年 5 月(本叙事截止)
**叙事原则**:真实经历,关键决策与转折点完整保留

---

## 引言:为什么有这个文档

很多项目讲起来是"我想做 X,然后我做了 X"。但工程项目的真实诞生过程往往是另一种形态——**你以为要做 X,做完发现是 Y,推翻一半重做才到 Z**。

Strata 的诞生属于后者。这份文档记录的是真实的判断、错误、和校准过程——不是事后润色的"完美故事"。

---

## 第一阶段:v1 的尝试与发现

### 缘起

我做这个项目的最初动机很简单——我想要一个**真正属于自己的、能成长的个人 AI 助理**。

市面上已有的方案分两类:

- **闭源 SaaS**(Notion AI / Daylio 等):功能预设、数据在云端、不能为我个性化演化
- **开源 framework**(LangChain / 各种 agent 库):给开发者用的,不是面向终端用户的产品

我想要的是第三种:**本地部署、IM 入口、数据归自己、能力随我使用而生长**。

### v1 的实现

2026 年初,我自己写了一个完整方案——`personal-assistant-spec.md`,大约 2800 行的 spec 文档。核心设计包括:

- 自建 raw_events 表 + 业务表系统
- 双 runtime 架构(Claude Agent SDK + PydanticAI)
- 五层数据架构
- 协议化人机协作循环
- 飞轮机制(让能力自我浮现)

我**完整实现并运行**了这套方案——核心数据流跑通,日常使用了一段时间,确认设计原则可行。

### 发现的两个问题

但用了一段时间后,我清楚地意识到两件事:

**问题 1:能力不齐全**

虽然核心数据流通了,但"会成长"这一块——也就是我最看重的能力——做得很初级。我当时设计的升级机制只是概念,**真正落地需要的代码生成、共建协议、回填机制都没有**。

**问题 2:实现成本比预期高**

我自己写了 raw_events 持久化、business 表 owner pipeline 隔离、向量检索、agent runtime 集成等等——这些**都是基础设施,不是我的研究价值所在**。继续往下做"能力浮现"的部分,需要的代码量会再翻几倍。

这两个问题让我停下来反思:**这个方向到底是不是对的?要不要重做?**

## 第二阶段:推翻重来的决定

### 关键判断

我决定重新审视整个项目,问几个核心问题:

1. **基础设施层** — 我自己造的 raw_events + agent runtime + memory 这些,**真的没有现成方案吗**?
2. **研究价值层** — 我真正想验证的东西是什么?
3. **如果别人已经做了 60% 的基础设施,我的真实工作量到底有多少**?

这次反思的结论是:**我之前的方案在"研究价值"和"基础设施"之间没有划清边界,导致大量精力花在了重复造轮子上**。

需要重做的不是代码,**是产品定位本身**。

## 第三阶段:重新调研生态

我开始系统性调研——这次不是"找灵感",而是**精确确认每一层有没有现成方案**。

### 调研维度

按"演化对象"把生态分成 4 个光谱:

| 演化对象 | 已有项目 |
|---|---|
| Meta-level(agent 自己变强) | capability-evolver, self-evolve, Letta Skill Learning, Hermes Agent |
| Data-level(数据自动结构化) | Mem0, Graphiti, ByteRover |
| Knowledge-level(知识自动组织) | memory-wiki (OpenClaw bundled), Cognee |
| **Domain-capability-level(用户业务能力浮现)** | **空白** |

**这是第一个关键发现**:

> 我想做的事(从用户输入流浮现完整业务能力——schema + 代码 + dashboard)**没有人做**。但相邻赛道全都很热,意味着这个空白是真空白,不是"伪需求"。

### 重要的几个调研结论

**OpenClaw**(我之前用过的 agent runtime):

通过实证 spike 验证——发"今天聊了 Lily / 妈妈 / 东野圭吾的小说"4 条消息,看 OpenClaw 默认行为:**消息进了 daily log,但没有自动结构化、没有写 wiki**。

让 OpenClaw agent 自己解释为什么没做,它的回答揭示了 OpenClaw 的设计哲学:**"轻量记忆范式——除非用户反复搜索,否则不固化"**。

这跟个人长期数据的场景**不匹配**——用户的消费记录、健康数据从第一次出现就是长期资产,不需要被搜索 3 次才被认可。

**Graphiti**(图知识引擎,看起来跟我的"业务表演化"高度相似):

做了三层评估:
- 第一层(能力):自动 entity + edge 抽取 + temporal validity,看起来完美
- 第二层(问题域):核心是**多跳语义推理**——但我的核心是**流水数据沉淀**,**两者问题域不重合**
- 第三层(成本):每次 add_episode 触发 5-10 次 LLM 调用,占月预算 37%——而且是固定成本

加上 OpenClaw 集成插件采用 session-bound 触发机制,跟"event-stream 哲学"错配——**最终决定不引入**。

**Hermes Agent**(NousResearch 出品,从 OpenClaw 演化来):

最接近我的位置。核心能力:Autonomous Skill Creation + 自演化机制 + 用户建模(Honcho 集成)。

仔细对比后差异清晰:
- Hermes 演化 **agent 自己**(skill markdown 文档)
- 我演化 **用户的业务系统**(可执行的 schema + 代码 + dashboard)

**两个相邻但本质不同的方向**——Hermes 让 agent 越用越懂你,我让你拥有越用越精准的私人软件平台。

### 调研对认知的校准

调研结束后,我对自己的位置有了完全不同的理解:

| 之前以为 | 实际发现 |
|---|---|
| 要自己造很多基础设施 | 90% 基础设施有现成方案 |
| 研究价值在"raw events + 业务表" | 研究价值在"协议层"——决定何时升级、如何升级 |
| 实施量 5000+ 行 | 核心研究层 1500-2000 行就够 |
| 我在做新东西 | 我在填一个被多个相邻项目暗示但没人走的空白 |

## 第四阶段:核心协议的设计

调研清晰后,我开始系统设计**真正属于研究价值的那一层**——能力生命周期协议。

### 几个关键问题逼出来的设计

设计过程中,几个问题对我影响很大,因为它们让我**否决了自己的某些假设**:

#### 问题 1:"我们这套不也是要多次出现才升级吗?那跟 OpenClaw lazy 范式有啥区别?"

这个问题让我意识到我之前用"lazy vs eager"对比是**错的**——我跟 OpenClaw 都是累积型,真正的差异是:

- **OpenClaw 用 search 信号触发升级**(消息被搜 3 次才固化)
- **我用 input pattern 信号触发升级**(消费这种类型反复出现就该浮现 schema)

**信号通道不同,升级形态不同**——这才是真正的差异。

#### 问题 2:"光升级不够,会不会出现 schema 爆炸?"

这让我加入了**双向演化协议**——升级跟降级是同构的协议,只是信号方向不同:

- 累积信号超阈值 → 升级
- 衰减信号超阈值 → 降级 / 归档

**没有降级机制的成长系统,长期看一定崩溃。**

#### 问题 3:"agent 输入的对话粒度跟数据粒度不一致怎么办?"

这是把理论变成代码时撞到的第一个真问题。

用户在一个 session 里说"今天买了 ¥45 咖啡 → 哦不对 ¥48 → 还买了司康 ¥18 → 总共 ¥66 → 记一下"——**6 条消息,但只对应 1 个语义事件**。

如果每条消息都当 raw_event,数据库里全是冗余和修正,飞轮分析时痛苦。

这逼出了**双层数据结构**:

- `messages` 表:每条 IM 消息原文,永不丢失
- `raw_events` 表:语义事件单位,有状态机(pending/committed/superseded/abandoned)

**raw_event 的语义不是"用户说了什么",而是"用户授权系统把什么当成事实"**。

#### 问题 4:"schema 升级了,历史数据怎么办?"

最开始记消费只分大类(food/goods),后期发现需要细分(meat/beverage/snack)——这时候历史 200 条数据怎么办?

这个问题逼出了**Schema 演化与回填协议**:

- 每张业务表带 `extraction_version` 字段
- Schema 变更登记到 `schema_evolutions` 表
- 后台 worker 异步回填(基于 messages 表重新抽取)
- 用户能看到回填进度和成本

**没有回溯能力,整个 lifecycle 协议的可信度就崩了**——这是从问题里逼出的核心设计。

#### 问题 5:"协议设计好了,但 schema 改变这件事**具体怎么发生**?"

这是最后一块——**Micro-app 全过程开发协议**。

我设计了三段式流程:

1. **Plan Phase** — 用户和 agent 对话精炼需求
2. **Spec Decomposition** — 拆成多个 OpenSpec atomic changes
3. **Orchestrated Build** — Orchestrator 依次调度 Claude Code 跑每个 change

这里有个**反复调整的关键决策**:

我最开始想自己写"Planner Agent + Build Phase A/B/C/D 切分",但后来发现 **OpenSpec(workflows profile)的现成 slash command 已经覆盖了 90%**:

- `/opsx:explore` → Plan Phase
- `/opsx:propose` → 自动生成 proposal/specs/design/tasks
- `/opsx:apply` → 执行 tasks
- `/opsx:verify` → 独立 validation gate
- `/opsx:archive` → 归档并 merge spec
- `/opsx:sync` → 反向同步 spec 跟代码

**又一次"不重复造轮子"的修正**——我留下的只有 Orchestrator 调度逻辑(500 行左右),其他全用 OpenSpec 现成机制。

### 协议套件:从五件简化到这五件

到最后,核心研究产出是 5 件协议(每一件都从具体场景里逼出来的,不是先有框架再填):

1. **能力生命周期状态机** — 三张表协作(proposals 提议 → builds 构建 → capability_registry 上线)
2. **业务能力浮现协议** — Reflect Agent 扫数据浮现"建议建能力"提议
3. **数据粒度协议** — 双层结构 messages/raw_events + 状态机
4. **Schema 演化与回填协议** — 升级后历史数据怎么跟着演化
5. **Micro-app 全过程开发协议** — 提案如何实际变成可执行代码

#### 两次自我修正:从 L0-L5 到三张表协作

设计早期我曾用过一个"L0-L5 渐进升级阶梯"的抽象——听起来很 AI、很研究。
后来真的开始想"L1 在数据库里长什么样、谁来写、什么时候转 L2"时,发现:

- 用户主动共建路径会直接跳到 active 状态,绕过中间级
- L1/L2 在数据流里没有具体对应物(raw_event 已经有 event_type,跟所谓"L2 domain identified"重复)
- 每级阈值都是凭直觉,没有 dogfood 数据支撑
- 简单的状态机已经完整覆盖了真实需求

**第一次修正**:砍掉 L0-L5,改成 6 状态的状态机(proposed → approved → building → active → archived → deleted)。

但写 SPEC 时又发现一个问题:**前 3 个状态在 `capability_registry` 表里永远不会出现**——因为提议阶段的状态由 `proposals` 表跟踪,构建阶段的状态由 `builds` 表跟踪,只有真正上线的 capability 才会进 `capability_registry`。塞进一张表的多状态字段会有 status='proposed' 但表里实际没数据这种尴尬。

**第二次修正**:改成三张表协作。proposals 管"提议",builds 管"构建过程",capability_registry 管"已上线"。每张表表达生命周期的一个阶段。

**两次修正都是同一个方向:简化、贴合实际工程**。

这是这次重做学到的最重要的一件事:**优雅 ≠ 必要**。一个 v1 时期不假思索带过来的概念,在 v2 实施推演时被自己识破——这种 self-correction 比"我设计了一套精美的多级阶梯"更值得讲。

前 4 件是**协议**,第 5 件是**让协议变成代码的发动机**。

## 第五阶段:产品定位最终落定

经过几轮调研和设计后,产品定位精确化了——

### 一句话定位

> **"一个本地部署的、IM 入口的、为单个用户在他自己机器上现场孵化业务能力的个人软件平台。"**

每个限定词都是差异点:

- "本地部署" → 不是 SaaS(vs Lovable / v0)
- "IM 入口" → 不是开发工具(vs Cursor / Claude Code)
- "为单个用户" → 不是开发者给别人造产品
- "现场孵化" → 不是预制功能(vs Notion AI / 记账 App)
- "业务能力" → 不是 agent 认知(vs Letta / Hermes)
- "个人软件平台" → 不是单一工具

### 在演化光谱里的位置

```
[meta-level evolution] — agent 自己变强
  capability-evolver / Hermes Agent / Letta Skill Learning

[data-level evolution] — 数据自动结构化
  Mem0 / Graphiti / ByteRover

[knowledge-level evolution] — 知识自动组织
  memory-wiki / Cognee

[domain-capability-level evolution] — 用户业务能力浮现
  ← Strata 在这里(生态空白)
```

### 30 秒版讲法

> "现在 AI 编程工具有两种范式:一种是 Lovable / v0,开发者描述需求 → AI 生成 SaaS 给别人用;另一种是 Cursor / Claude Code,AI 辅助开发者写自己的代码。
>
> 但有第三种范式还没人做:**终端用户用自然对话,AI 在他自己机器上为他一个人长出专属软件**。
>
> 我做的就是这个。具体是个本地部署的 IM 助理——用户用 Telegram 跟它聊,聊久了系统识别出'这个用户在追踪消费',自动提议建立完整的消费追踪能力(schema + 代码 + dashboard),用户审核后调用 Claude Code 作为内部组件真的把代码写出来集成进系统。
>
> 跟现有所有方案的根本差异是:数据归用户、能力归用户、演化由用户和系统共同决策、最终产物是一个'随你的生活生长'的个人软件平台。"

## 第六阶段:工程实现的最终架构

设计稳定后,最终实施分工:

### 现成实现层(零工作量)

| 功能 | 谁提供 |
|---|---|
| IM 入口 | OpenClaw Telegram channel |
| Agent runtime | OpenClaw |
| Memory tier(短期/recall) | memory-core |
| Wiki 容器 | memory-wiki bundled |
| Model provider 抽象(用户的 API key / OAuth / 订阅) | OpenClaw model system |
| 代码生成 | Claude Code CLI subprocess |
| Spec-driven 工作流 | OpenSpec workflows profile |
| 向量 / 全文检索 | sqlite-vec + FTS5 |

### 研究价值层(自己写,~1500-2500 行 TypeScript)

| 模块 | 职责 |
|---|---|
| Triage | 意图分类(capture / query / build_request) |
| Tools + Skills | raw_events 状态机管理 |
| Reflect Agent | 扫数据浮现升级 / 降级提议 |
| Build Bridge | Plan Phase → Decompose → Orchestrate → Integrate 全流程 |
| Orchestrator | 调度 Claude Code 跑 OpenSpec changes |
| Integration Layer | 接收产物集成回运行时 |
| Re-extraction Worker | 历史数据回填 |

**核心写的代码 100% 是研究价值层 + 必要的桥接,没有一行重复造轮子**。

### 项目最终定名:Strata

中文是"地层 / 沉积岩"——这个名字精确捕捉了产品的本质:

- 用户的数据像沉积岩一样层层堆积(messages → raw_events → 业务表 → wiki)
- 能力像地层中的化石一样,从沉淀的数据里浮现出来
- 时间维度天然在场——古老的地层(历史数据)依然在,新的地层(新能力)持续生成

## 阶段总结与反思

### 整个过程的时间线

| 时间 | 阶段 | 产出 |
|---|---|---|
| 2026 年初 | v1 想法 + 实现 | 2800 行 spec + 跑通核心数据流 |
| 2026 年 4 月底 | 发现 v1 问题 | 决定推翻 |
| 2026 年 5 月初 | 重新调研生态 | 多个项目精确对比 |
| 2026 年 5 月 5-7 日 | OpenClaw 实证 spike | 4 条小样本验证发现 |
| 2026 年 5 月 8-10 日 | 协议设计与多轮校准 | 五件套协议成型 |
| 2026 年 5 月 11-12 日 | 工程契约文档 | 4000+ 行设计文档 |

### 几个值得记下的判断时刻

**判断 1:推翻自己的 v1**

最难的不是写新方案,是**承认旧方案要废**。但 v1 已经显示了"实施成本会超过研究价值"——再继续就是沉没成本谬误。

**判断 2:不引入 Graphiti**

Graphiti 在技术上很 sexy,跟我的设计看起来高度契合。**第一直觉是"必用"**。但三层评估后发现核心问题域不重合 + 架构哲学错配——**否决了一个看起来完美的方案,需要工程定力**。

**判断 3:不重复造 Planner Agent**

我最开始设计了自己的 Planner Agent。后来发现 OpenSpec 的 `/opsx:propose` 已经做了这件事——**主动取消自己的设计,改用现成方案**。

**判断 4:接受"渐进交付"代替"一次跑通"**

最开始想让 Claude Code 一次性生成完整 micro-app。后来意识到一次性的 headless 跑通率只有 40-60%,而拆成多个 atomic change 跑可以 >90%。**牺牲速度换可控性,是对的取舍**。

### 这次经历的核心收获

如果说从这次"推翻 v1、重做 v2"的经历里学到一件事——

> **判断一个开源项目是否值得引入的标准,不是"它能做什么",而是"它解决的核心问题跟我产品的核心需求有多重合"。**

很多项目能做的事跟你 99% 重合,但**核心问题**可能完全不同。引入它就是把别人的复杂度搬到你的产品里,而真正用上的只是它能力的 1%——这种代价往往隐性、长期、致命。

Strata 现在的架构上,**每个引入的依赖都有"为什么必须用它"的清晰回答**。这是 v1 没做到的——也是 v1 实施成本失控的根本原因。

## 几个面试时可能被追问的问题(及思考方向)

### Q: "推翻 v1 重做,是不是浪费时间?"

不是浪费。v1 让我知道了:
1. 核心数据流确实能 work(降低了 v2 的不确定性)
2. 自己造基础设施成本会失控(避免了 v2 重蹈覆辙)
3. 真正的研究价值在协议层,不在数据层(精确化了产品定位)

**这些认知没有 v1 的实施经验,光靠"想"想不出来**。v1 是必要的失败。

### Q: "你这套设计真的能 work 吗?有没有验证?"

部分验证:
- OpenClaw 实证 spike 跑过,4 个发现都是源码 / agent 自述级证据
- Graphiti 三层评估有源码 + GitHub Issue 证据(社区在等同样的能力)
- 五件套协议各自的设计选择都有具体场景驱动

完整端到端验证需要 Phase 0 实施。当前计划是先用 ~4 周做最小端到端骨架(messages 表 + capture 流程 + 一个手动 capability),再用 ~4 周补 Build Bridge / Reflect Agent。但考虑到 v1 用了 4 个月,这个时间表可能过于乐观——以最小骨架为硬承诺,完整功能视实施情况调整。

### Q: "怎么知道这个方向不是 6 个月后就被 Hermes 吃掉了?"

Hermes 当前路线是"self-improving coding agent"——演化对象是 agent 自己。

要吃掉 Strata 的赛道,NousResearch 需要做战略转向:从"让 agent 学习"转到"为用户长 micro-app"——**这是个产品定位级的转向,不是技术升级**。

我估计 6-12 个月的窗口。需要在这个时间内把"业务能力浮现协议"走深,做到即使他们转向也追不上的位置。

### Q: "你为什么相信用户真的需要这个?"

调研里有个意外发现——一个叫 PennyTrail 的个人项目用 Telegram bot + n8n + Supabase **DIY 拼了一个个人记账系统**,博客访问量不低。

这种 DIY 拼凑本身就是市场信号——**用户在用自己的方式凑出类似的产品形态**,但没有一个通用方案。我做的事情是把这种 DIY 范式产品化、扩展到所有领域。

### Q: "如果让你重新走一遍,会更快吗?"

会。我会:
1. **更早做生态调研**——v1 之前就花一周做调研,可能就不会自己造那么多轮子
2. **更早做实证 spike**——比起读文档,跑一次 spike 看到的东西多得多
3. **更早暴露设计 trap**——比如"raw_event 粒度跟对话粒度对不齐"这种问题,如果在 v1 就想清楚,可以省很多重构

但也有一些东西**只能通过 v1 才能学会**——比如"自己造基础设施成本会失控"这种判断,光靠想是不会信服的。

## 文档结束

这就是 Strata 真实的诞生过程。**没有润色,没有事后修饰**——包括我推翻自己方案、被反问"不也是 lazy 范式吗"的尴尬瞬间、被一个开源项目逼着重新审视自己研究空白的几次校准。

工程项目的诞生从来不是直线——是反复撞墙、修正、再撞墙、再修正的迭代过程。这就是为什么 Strata 这个名字最后落到"地层 / 沉积"上——**不只产品的内涵是沉积,设计过程本身也是沉积**。
